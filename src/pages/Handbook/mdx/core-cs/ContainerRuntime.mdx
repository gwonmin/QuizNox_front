## Kubernetes와 Container Runtime 관계

**Pod**는 Kubernetes가 컨테이너를 묶어서 배치하는 최소 단위입니다. (한 Pod 안에 컨테이너 하나 또는 여러 개가 들어갈 수 있다.)

**한 줄 흐름:** Control Plane이 Pod를 노드에 스케줄하면, 해당 노드의 **kubelet**이 **CRI**를 통해 런타임을 호출해 이미지 풀 → Sandbox(Pod 네트워크·PID 네임스페이스) → 컨테이너 생성·시작 순으로 실행한다.

## CRI (Container Runtime Interface)

Kubernetes가 다양한 런타임(containerd, CRI-O 등)을 플러그인처럼 쓰기 위한 표준 인터페이스다. gRPC 기반이며, **ImageService**(이미지 풀·목록·삭제)와 **RuntimeService**(Pod Sandbox·컨테이너 생성·시작·중지·상태 조회)로 나뉜다.

<KeyTable
  columns={["CRI 서비스", "주요 역할"]}
  rows={[

    ["RuntimeService", "RunPodSandbox( Pod용 네트워크·PID 네임스페이스 ), CreateContainer, StartContainer, StopContainer 등. 실제 컨테이너 생명주기 제어."],

## OCI와 런타임 구현

런타임이 따르는 표준으로 **OCI(Open Container Initiative)** 가 있다. **이미지 스펙**(레이어·매니페스트 포맷)과 **런타임 스펙**(파일시스템 번들·실행 규격)으로 나뉘며, containerd·CRI-O는 이 스펙을 구현해 Kubernetes CRI와 연결된다. Docker 이미지는 OCI 호환 형식으로 변환해 사용할 수 있다.

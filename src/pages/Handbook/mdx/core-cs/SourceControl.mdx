### Git Flow

<MermaidDiagram
  code={`gitGraph
  commit id: "init"
  branch develop
  commit id: "feat-setup"
  branch feature/login
  commit id: "login-ui"
  commit id: "login-api"
  checkout develop
  merge feature/login id: "merge-login"
  branch release/1.0
  commit id: "bump-version"
  checkout main
  merge release/1.0 id: "v1.0.0" tag: "v1.0.0"
  checkout develop
  merge release/1.0 id: "sync-release"
  checkout main
  branch hotfix/1.0.1
  commit id: "fix-bug"
  checkout main
  merge hotfix/1.0.1 id: "v1.0.1" tag: "v1.0.1"
  checkout develop
  merge hotfix/1.0.1 id: "sync-hotfix"`}
/>

> **main**은 항상 배포 가능한 상태, **develop**에서 기능을 통합하고, **release** 브랜치에서 릴리스를 준비한다. 긴급 수정은 **hotfix**로 main에서 직접 분기한다.

### Trunk-based Development

<MermaidDiagram
  code={`gitGraph
  commit id: "init"
  branch feat-A
  commit id: "A-wip"
  checkout main
  merge feat-A id: "merge-A"
  commit id: "ci-fix"
  branch feat-B
  commit id: "B-wip"
  checkout main
  merge feat-B id: "merge-B"
  branch feat-C
  commit id: "C-wip"
  checkout main
  merge feat-C id: "merge-C" tag: "v1.1.0"`}
/>

> **main 하나**를 중심으로 짧은 수명의 브랜치를 빠르게 머지한다. 머지 빈도가 높아 충돌이 줄고, main 푸시마다 CI/CD가 실행된다.

### GitHub Flow

<MermaidDiagram
  code={`flowchart LR
  A["main"] -->|branch| B["feature branch"]
  B -->|commit & push| C["PR 생성"]
  C -->|CI 통과 + 리뷰| D["main에 머지"]
  D -->|자동 배포| E["Production"]

  style A fill:#dbeafe
  style E fill:#d1fae5`}
/>

> main에서 브랜치를 만들고, PR을 통해 리뷰·CI를 거쳐 main에 머지하면 곧바로 배포된다. 가장 단순한 전략이다.

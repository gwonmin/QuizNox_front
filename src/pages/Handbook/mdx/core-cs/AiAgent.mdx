## ReAct 패턴 (Reasoning + Acting)

AI Agent의 핵심 실행 패턴으로, **추론(Thought)과 행동(Action)을 번갈아 수행**하며 **관찰(Observation) 결과를 다음 추론에 반영**한다.

<MermaidDiagram
  code={`flowchart LR

  T1["Thought 1\n(무엇을 해야 하는가)"]
  A1["Action 1\n(도구 호출)"]
  O1["Observation 1\n(결과 확인)"]
  T2["Thought 2\n(다음에 무엇을 해야 하는가)"]
  A2["Action 2\n(추가 도구 호출)"]
  O2["Observation 2\n(결과 확인)"]
  T3["Thought 3\n(답변 가능)"]
  ANS["최종 답변"]

  Q --> T1 --> A1 --> O1 --> T2 --> A2 --> O2 --> T3 --> ANS`}

## LangChain / LangGraph 심화

LangChain은 **부품(모듈)**, LangGraph는 그 부품을 **조립하는 설계도(그래프)**이다. 이 관계를 단계적으로 풀어본다.

### 1단계: 부품 이해 — LangChain 핵심 모듈

LangChain은 LLM 애플리케이션에 필요한 모듈을 제공하고, 이들을 파이프(`|`) 연산자로 연결한다.

### 2단계: 조립 — LangGraph로 Agent 워크플로우 구성

단순 체인(`A → B → C`)은 직선형이라 **"도구를 쓸지 말지 LLM이 판단"** 하는 루프를 표현할 수 없다. LangGraph는 **그래프(노드 + 조건부 엣지)** 로 이 문제를 해결한다.

> 핵심은 **agent ↔ tools 루프**이다. LLM이 매 회차마다 "어떤 도구를 호출할지" 또는 "답변을 완료할지"를 **자율적으로 판단**한다. 3회차에서 Retriever Tool을 호출한 것이 RAG가 Agent 안에서 **도구 중 하나로 작동**하는 대표적인 패턴이다. 이 루프 구조가 LangGraph가 아닌 단순 체인으로는 불가능한 이유이다.

> 이 아키텍처의 핵심은 **Agent Runtime 내부의 라우터 패턴**이다. 모든 요청을 동일한 Agent 루프로 처리하는 대신, 의도 분류 후 단순 질의는 RAG 노드로, 작업 요청은 Action 노드로, 복합 작업은 Planner를 거쳐 처리한다. 이렇게 하면 불필요한 도구 호출을 줄이고 응답 속도와 비용을 최적화할 수 있다.

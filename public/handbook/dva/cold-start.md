# Lambda Cold Start

**첫 호출·오래 비사용 후** 런타임 초기화로 인한 지연입니다. 지연 민감 API·동기 호출에서 프로비저닝 동시성·패키지 최소화로 완화합니다.

---

## 1. 특징

- **원인**: 실행 환경(인스턴스)이 없을 때 **새 인스턴스**를 할당하고 런타임·초기화 코드(핸들러 밖 전역 코드) 실행. 이 구간이 Cold Start 지연.
- **완화 1 – Provisioned Concurrency**: 미리 워밍된 인스턴스 수를 유지해 첫 요청부터 짧은 지연.
- **완화 2 – 패키지·초기화**: 배포 패키지·레이어 크기 축소, 필요한 SDK만 포함, **SDK·DB 연결 등은 핸들러 밖(전역)**에 두어 재사용 시 초기화 비용 제거.
- **메모리**: 메모리 증가 시 CPU 비율도 증가 → CPU 바운드 작업 지연 감소(콜드 스타트 자체보다는 실행 시간 단축).

---

## 2. 시나리오

| 조건 | 선택 |
|------|------|
| API Gateway 동기 호출·지연 민감 | Provisioned Concurrency로 워밍 유지 |
| 배포 패키지 과대·초기화 느림 | Lambda Layer로 의존성 분리, 불필요 라이브러리 제거 |
| DB 연결 매 호출 생성 | 연결·Statement를 핸들러 밖 전역에서 생성·재사용; RDS는 RDS Proxy 권장 |
| Lambda 임시 파일(10MB 이하·호출 내만 사용) | **/tmp** 사용(최대 512MB); 호출 종료 후 삭제됨. EFS·S3는 공유·영구 저장 시. |
| 새 버전 배포 후 즉시 트래픽 받음 | 새 버전에 대해 Provisioned Concurrency 설정 후 트래픽 전환 |

---

## 기출빈출

| 시나리오 | 접근 |
|----------|------|
| CPU 바운드·실행 시간 단축 | Lambda **메모리** 증가(CPU 비율 동반 증가). |
| 지연 민감·동기 API 콜드 스타트 완화 | **Provisioned Concurrency**로 워밍된 인스턴스 유지. |
| 초기화·지연 감소 | **SDK·DB 연결**을 핸들러 **밖(전역)**에서 초기화·재사용. |
| RDS 연결 수 과다 | **RDS Proxy** 사용; 연결을 핸들러 밖에서 재사용. |
| 여러 Lambda가 공통 라이브러리·의존성 공유 | **Lambda Layer**로 공통 코드 분리 → 배포 패키지 축소·관리 용이. |

## 요약

| 항목 | 설명 |

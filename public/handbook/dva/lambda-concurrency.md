# Lambda Concurrency

**동시에 실행되는 인스턴스 수**를 제어하는 개념입니다. 예약 동시성·프로비저닝 동시성으로 트래픽 격리·콜드 스타트 완화·다운스트림 보호가 가능합니다.

---

## 1. 특징

- **동시 실행(Concurrency)**: 요청마다 실행 환경(인스턴스)이 할당되며, 계정 기본 한도(동시 실행 수)와 **함수별 예약 동시성(Reserved Concurrency)**으로 제한됨.
- **예약 동시성(Reserved Concurrency)**: 해당 함수가 쓸 수 있는 **동시 실행 상한**. 다른 함수와 격리·스로틀링으로 서드파티 API·DB 연결 수 보호.
- **예약 0**: 해당 함수 호출 불가(비활성화용).
- **프로비저닝 동시성(Provisioned Concurrency)**: 미리 워밍된 인스턴스 수 유지 → Cold Start 감소, 지연 민감 워크로드에 사용.
- **SQS 이벤트 소스**: **이벤트 소스별 최대 동시성(maximum concurrency)**으로 서드파티 rate limit에 맞춰 Lambda 호출 수 제한 가능.

---

## 2. 시나리오

| 조건 | 선택 |
|------|------|
| 서드파티 API 호출 수 제한 준수 | 예약 동시성 또는 SQS 이벤트 소스 최대 동시성으로 Lambda 동시 호출 수 제한 |
| Cold Start 제거·예측 가능 지연 | 프로비저닝 동시성 설정 |
| 특정 함수만 격리·상한 두기 | 해당 함수에 예약 동시성 설정 |
| 함수 비활성화 | 예약 동시성 0 |
| 스로틀(Throttled) 증가 시 | 예약 동시성 상향 또는 서비스 할당량 증가 요청 |

---

## 기출빈출

| 시나리오 | 접근 |
|----------|------|
| RDS "too many connections" | **RDS Proxy**로 연결 풀링; Lambda는 Proxy 경유로 접속. |
| Lambda 스로틀·실패 | **예약 동시성** 상향 또는 **DLQ**로 실패 이벤트 수집·재처리. |
| 서드파티 API rate limit 준수 | **SQS 이벤트 소스 최대 동시성** 또는 함수 **예약 동시성**으로 호출 수 제한. |
| Lambda 스로틀 완화(운영 효율) | **예약 동시성** 증가 또는 **서비스 할당량 증가** 요청. |

## 요약

| 항목 | 설명 |
|------|------|
| 동시 실행 | 요청당 인스턴스 할당, 계정·함수별 한도 존재 |
| 예약 동시성 | 함수별 상한·격리·다운스트림 보호 |
| 프로비저닝 동시성 | 워밍 유지 → Cold Start·지연 완화 |
| 조건 | "rate limit·연결 수 제한" → 예약/최대 동시성 / "지연 민감" → Provisioned Concurrency |

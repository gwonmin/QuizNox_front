# Scale-up vs Scale-out

**용량을 늘리는 두 방식**입니다.  
Scale-up = 수직 확장, Scale-out = 수평 확장.

## Scale-up (수직 확장)

- **한 서버**의 CPU·메모리·디스크를 키움
- 장점: 구조 단순, 애플리케이션 변경 거의 없음
- 단점: 하드웨어 상한, 단일 장애점(SPOF), 스펙 올릴수록 비용이 급격히 증가

## Scale-out (수평 확장)

- **서버 개수**를 늘림
- 장점: 상한 완화, 장애 시 일부만 영향, 비용이 대체로 선형
- 단점: 공유 상태·세션·데이터 일관성·로드밸런싱 설계 필요

## 개념 도식

```mermaid
flowchart TB
  subgraph up [Scale-up]
    A[서버 1대]
    A --> B[동일 서버 스펙 업]
    B --> C[더 큰 1대]
  end
  subgraph out [Scale-out]
    D[서버 1대] --> E[서버 2대]
    E --> F[서버 N대]
  end
```

## 실제 예시

| 구분 | 예시 | 이유 |
|------|------|------|
| Scale-up | DB 서버 메모리·CPU 증설 | 단일 인스턴스 성능을 올려야 할 때. 대수를 늘리면 데이터 분할·일관성이 복잡해질 수 있음. |
| Scale-out | 웹 서버·API 서버 여러 대 | Stateless라면 대수만 늘리면 됨. 로드밸런서 뒤에 인스턴스를 추가. |

## 요약

| 구분 | Scale-up | Scale-out |
|------|----------|-----------|
| 대상 | 한 노드 스펙 | 노드 개수 |
| 한계 | 하드웨어 상한 | 설계·일관성·비용 |
| 적합 | 단일 노드로 충분할 때 | 트래픽·장애 분산이 필요할 때 |

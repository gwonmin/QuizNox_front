# SQS · SNS 디커플링 및 알림

**SQS**는 **비동기 메시지 큐**로 컴포넌트 간 **디커플링**과 **버퍼** 역할을 하고, **SNS**는 **Pub/Sub**로 한 이벤트를 여러 구독자에게 알립니다. 고가용성·확장 가능한 이벤트 기반 아키텍처에 자주 조합됩니다.

---

## 특징

### SQS (Simple Queue Service)

- **역할**: 생산자·소비자 분리, 메시지 큐로 비동기 처리
- **시나리오**: 처리 부하 완화, 장애 시 재시도, 워크로드 버퍼
- **표준 vs FIFO**: 표준(최소 1회 전달·높은 처리량), **FIFO**(정확히 1회·**순서 보장**) → "주문 순서 유지", "처리 순서 필수" 시 FIFO
- **Visibility Timeout**: 메시지 수신 후 처리 완료 전에 다른 소비자가 같은 메시지를 받지 않도록 함. **너무 짧으면** 처리 중 메시지가 다시 노출되어 **중복 처리** 발생 → 적절히 늘리거나 처리 완료 시 삭제·visibility 연장
- **ASG 연동**: 스케일링 정책을 **큐 메시지 수(ApproximateNumberOfMessagesVisible)** 기준으로 하면 작업량에 맞춰 워커 수 자동 증감

---

### SNS (Simple Notification Service)

- **역할**: 토픽 발행 → 구독자(이메일·SQS·Lambda 등)에게 푸시
- **시나리오**: 알림, 이벤트 팬아웃, 여러 시스템에 동시 전달
- **SQS와 조합**: SNS → SQS로 팬아웃 후 각 큐에서 순차 처리
- **내구성**: SNS만 쓰면 구독자(Lambda 등) 장애 시 메시지 유실 가능. **SNS → SQS → Lambda**로 하면 큐가 메시지 보관·재시도·DLQ로 실패 처리 가능 → "네트워크 일시 장애에도 모두 처리" 시 SQS 구독

---

## 시나리오

| 요구 | 선택 |
|------|------|
| 작업 큐·디커플링 | SQS |
| **수신 순서대로 처리 필수**(예: 주문) | **SQS FIFO** (API Gateway → SQS FIFO → Lambda) |
| 한 이벤트를 여러 대상에 전달 | SNS |
| 이벤트 + 큐로 버퍼링·재시도 | SNS → SQS (→ Lambda) |
| 큐 길이에 따라 워커 수 자동 조절 | ASG 스케일링 정책 = SQS 메시지 수 |
| 중복 처리 방지 | Visibility Timeout 적절 설정·처리 완료 시 삭제 |

---

## 기출빈출

| 시나리오 | 접근 |
|----------|------|
| 수신 순서대로 처리 필수(주문 등) | **SQS FIFO** (API Gateway → SQS FIFO → Lambda). |
| 이벤트 + 버퍼·재시도·내구성 | **SNS → SQS → Lambda** (SNS만 쓰면 구독자 장애 시 유실 가능). |
| 큐 길이 기준 워커 자동 조절 | ASG 스케일링 정책 = **ApproximateNumberOfMessagesVisible**. |
| Visibility Timeout 너무 짧음 | 처리 중 메시지 재노출 → **중복 처리**; 늘리거나 처리 완료 시 삭제. |

## 요약

| 항목 | 설명 |
|------|------|
| SQS | 메시지 큐, 디커플링·비동기 처리; FIFO = 순서 보장 |
| SNS | Pub/Sub, 알림·팬아웃; 내구성 필요 시 SNS→SQS→Lambda |
| Visibility Timeout | 짧으면 중복 처리 가능 → 조정 또는 처리 완료 후 삭제 |
| 조건 | "순서 보장" → SQS FIFO / "재시도·내구성" → SNS→SQS / "큐 기반 스케일" → ASG·SQS |

# DynamoDB 사용 케이스

**키-값·문서형** NoSQL로 적합한 워크로드입니다. 키 기반 조회·수평 확장·저지연이 강점이고, 복잡한 조인·전체 스캔이 주면 RDS/Aurora를 고려합니다.

---

## 특징

- **적합**: 키 기반(파티션 키·정렬 키) 조회가 주인 경우, 수평 확장·스키마 유동적, 짧은 지연이 중요한 경우
- **부적합**: 복잡한 조인·집계, 전체 테이블 스캔이 주인 경우, 관계형 모델이 자연스러운 경우
- **처리량**: 프로비저닝 또는 온디맨드 모드로 읽기/쓰기 용량 설정. DAX로 캐싱 가능

---

## 시나리오

| 조건 | 선택 |
|------|------|
| 키 기반·저지연·스키마 유동 | DynamoDB |
| **트래픽 예측 어려움·급격한 스파이크** | **DynamoDB On-Demand** |
| **RPO·데이터 손상 복구** | **DynamoDB PITR** 활성화 |
| **장기 보존(예: 7년)·스케줄 백업** | **AWS Backup**으로 DynamoDB 백업·보존 정책 설정 |
| 복잡한 쿼리·조인·트랜잭션 | RDS·Aurora |
| 캐시로 읽기 부하·지연 감소 | DynamoDB + DAX 또는 ElastiCache |

---

## 기출빈출

| 시나리오 | 접근 |
|----------|------|
| 트래픽 예측 어려움·급증 | **DynamoDB On-Demand**. |
| RPO 짧음·데이터 손상 복구 | **PITR(Point-in-Time Recovery)** 활성화. |
| 장기 보존·스케줄 백업(예: 7년) | **AWS Backup**으로 DynamoDB 스케줄·보존. |

## 요약

| 항목 | 설명 |
|------|------|
| DynamoDB | 키 기반·수평 확장·저지연 NoSQL |
| On-Demand | 예측 어려운·급증 트래픽에 적합 |
| PITR | 짧은 RPO 복구; AWS Backup = 장기 보존·스케줄 |
| 조건 | "키 조회", "확장" → DynamoDB / "순간 스파이크" → On-Demand / "RPO" → PITR |
